## Bugs

### B1 · `configurable.ts:69` — symbol key 下类型断言不安全 ✅ 已修复（KAI）

```ts
// 修复前
Object.hasOwn(_presets as object, key as string)

// 修复后
Object.hasOwn(_presets, key as PropertyKey)
```

`keyof T` 包含 `string | number | symbol`，`key as string` 在 symbol key 场景下是错误的强制转型。`Object.hasOwn` 原生接受 `PropertyKey`，不需要强制转换。同时 `_presets as object` 也是多余的（`_presets` 已是 `T extends object`）。

### B2 · `cloneObject.ts` — 数组降级产生错误结构 ✅ 已修复（KAI）

`cloneObjectByJson` 在 `JSON.stringify` 失败时降级到 `cloneObjectByAssign`：

```ts
Object.assign({}, [1, 2, 3])
// → { '0': 1, '1': 2, '2': 3 }  ← 不是数组！
```

循环引用数组触发降级时，返回值静默地从 `T[]` 变成普通对象，违反函数签名 `<T extends object>(obj: T): T`。

**修复：** 降级前检测是否为数组，数组走 `[...obj]` 浅拷贝。

```ts
return Array.isArray(obj)
  ? ([...obj] as unknown as T)
  : cloneObjectByAssign(obj);
```

### A5 · `pending.ts` — Promise 构造器只有 resolve ⚠️ 还原

**原始建议**：改为 `Promise.resolve().then(...)` 模式提升可读性。

**KAI 实施后发现问题**：`Promise.resolve().then()` 是异步的，fn 被推进 microtask queue，三个并发调用全部进来、counter 累加完毕后 fn 才执行，导致 `getPendingCount()` 首次返回值从 0 变成 2，破坏了 `pendingFn` 的核心语义——"首次调用时等待者数为 0"。

**还原原因**：`new Promise((resolve) => { resolve(fn()) })` 的 executor 同步执行，fn 在 `inflight.set` 之前就跑，保证了 counter 的正确时序。错误处理同样完整：constructor 捕获同步 throw，`resolve(rejectedPromise)` 传播异步 reject。可读性问题通过注释说明即可，不值得牺牲行为正确性。
