### B1 · `configurable.ts:69` — symbol key 下类型断言不安全

```ts
// 当前
Object.hasOwn(_presets as object, key as string)

// 修复
Object.hasOwn(_presets, key as PropertyKey)
```

`keyof T` 包含 `string | number | symbol`，`key as string` 在 symbol key 场景下是错误的强制转型。`Object.hasOwn` 原生接受 `PropertyKey`，不需要强制转换。同时 `_presets as object` 也是多余的（`_presets` 已是 `T extends object`）。

### B2 · `cloneObject.ts` — 数组降级产生错误结构

`cloneObjectByJson` 在 `JSON.stringify` 失败时降级到 `cloneObjectByAssign`：

```ts
Object.assign({}, [1, 2, 3])
// → { '0': 1, '1': 2, '2': 3 }  ← 不是数组！
```

循环引用数组触发降级时，返回值静默地从 `T[]` 变成普通对象，违反函数签名 `<T extends object>(obj: T): T`。

**建议修复：** 降级前检测是否为数组，数组走 `[...obj]` 浅拷贝。

```ts
export const cloneObjectByJson = <T extends object>(obj: T): T => {
  _assertObject(obj);
  try {
    return JSON.parse(JSON.stringify(obj));
  } catch {
    console.warn('cloneObjectByJson: fallback to shallow copy');
    return Array.isArray(obj)
      ? ([...obj] as unknown as T)
      : cloneObjectByAssign(obj);
  }
};
```

### B4 · `decimalAdjust:209` — 多余的 typeof 检查（不影响正确性）

```ts
_exp = +_exp;  // 已经是 number 类型
// ...
!(typeof _exp === _typeNum && _exp % 1 === 0)
//  ^^^^^^^^^^^^^^^^^^^^^ 多余：+exp 之后必然是 number
```

`NaN % 1` 是 `NaN`，`NaN === 0` 为 false，`_exp % 1 === 0` 已经涵盖了 NaN 情况。可简化为 `!(_exp % 1 === 0)`。

## 架构隐患

### A1 · `path.ts` — `dangerReplace` 在 `join` 内部可能被多次调用

`join` 的 `handlePart(path)` 调用 `purge(path)`（触发 `dangerReplace`），若结果含分隔符则 `split` 后递归处理每个子段，每个子段再次调用 `purge` → `dangerReplace`。

对于同一个输入路径 `"../foo"` 拆成两个子段 `".."` 和 `"foo"`，`dangerReplace` 被调用 3 次（整体 1 次 + 子段 2 次）。若用户实现了有状态的安全回调（如攻击检测计数），行为可能出乎意料。

**建议：** 在文档中明确说明此行为，或在 `handlePart` 内区分"已 purge 的子段不再重复执行 dangerReplace"。

> 此处已修复，思源也进行检查，并帮忙修正了 JSDOC。

### A2 · `pending.ts` — 全局 `scopeRegistry` 是模块级副作用

`scopeRegistry` 是 `Set`，跨所有 `pending` / `pendingFn` 调用共享。测试中已用 `afterEach(clearPendingRegistry)` 处理，但以下场景仍需注意：

- 库被多处 import（ESM 缓存，同一实例，无影响）
- 动态 import 重新加载场景（scope 可能无法重新注册）

### A3 · `StatefulRpc.settle()` — 事件发射先于 Promise settle，但不等待

```ts
this.#emitter.emit(settled.type, ...).catch(console.error);  // fire-and-forget
this.#emitter.emit('settle', ...).catch(console.error);
if (settled.type === 'resolve') {
  it.resolve(settled.result);  // Promise settle 在此
}
```

事件处理器是异步的，Promise 的 `.then` 回调可能先于事件处理器执行完毕。若事件处理器中有依赖"结果尚未被消费"的副作用逻辑，会产生竞态。

> janpoem 已处理：
> 这里我已经处理了，events/index.ts ，emit 不会抛出错误，改为由 onError 进行接管，即事件 emit 时抛出的错误时，是静默的，这样调用的代码（反复写`emit().catch(console.error)` 这样确实很麻烦）。
> 需要完善 events/index.ts 的注释文档，说明 emit 的机制
> 经过和思源探讨，pending(addPendingItem) 和 settle 都改为，非阻塞，但重要时先执行，比如 timer 启动，或者 resolve or reject（先把 pending 结果输送出去，而后才 emit）
> 在 StatefulRpc 层面，事件只起通知作用，而不阻塞执行，这个也需要在注释文档强调，而非起拦截的作用。

### A4 · `EventsEmitter` — 缺少 `removeAllListeners`

`createEmitter` 没有提供批量清理的方法。长期存活的 emitter（如 `StatefulRpc` 实例）在组件频繁挂载/卸载的场景下，若调用方忘记调用返回的 unsubscribe 函数，监听器会持续积累。

**建议：** 添加 `removeAllListeners(name?: keyof E)` 方法，或提供 `dispose()` 清理全部。

> ```

### A5 · `pending.ts:129` — Promise 构造器只有 resolve

```ts
const promise = new Promise<Awaited<T>>((resolve) => {
  resolve(fn(params, ...args) as ...);
```

行为上正确（同步抛出被 Promise 构造函数捕获，`resolve(rejectedPromise)` 传播拒绝），但只写 `resolve` 不写 `reject` 的模式让代码意图不直观，容易让读者误认为漏掉了错误处理。建议改为显式的 `try/catch + reject`，或使用 `Promise.resolve().then(...)` 模式。
